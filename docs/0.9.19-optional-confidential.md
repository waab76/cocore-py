# 0.9.19 — optional, per-machine confidential upgrade

## Goal

Make "go confidential" an **optional, per-machine, owner-initiated** upgrade whose
entire effect is to republish that machine's `dev.cocore.compute.provider` record at
a higher `trustLevel`/`tier`. Machines that never opt in are byte-for-byte unaffected.
There is **no fleet cutover** — confidential is a property each machine earns and
publishes on its own record (invariant #1), not a global advisor switch.

## The model (already encoded in the lexicon)

Two orthogonal axes on the provider record:

- **`trustLevel`** (environment): `self-attested` → `hardware-attested`. Hardware-
  attested = a real Apple MDA chain bound to the signing key (genuine hardware +
  SIP). Does NOT require the native engine.
- **`tier`** (work): `best-effort` → `attested-confidential`. Attested-confidential =
  the measured native in-process engine + bound MDA + hardened posture + code
  identity. This is the only rung that makes the prompt private from the operator.

The honest ladder, each rung independently opt-in:

| Rung | trustLevel | tier | Earns it | Protects |
|---|---|---|---|---|
| default | self-attested | best-effort | nothing (subprocess) | transport only |
| light | **hardware-attested** | best-effort | MDA enrollment (the wizard) | genuine hw + SIP; **operator can still read prompts** |
| full | hardware-attested | **attested-confidential** | + native engine + code identity + known-good cdHash | operator cannot read the prompt |

## Current state (from the code map)

**Already wired:** the lexicon (`trustLevel` + `tier`); MDA chain loading
(`mda_loader.rs`, 3 strategies); native-engine gating (`build_engines` + env);
attestation `tier` computation (`attestation.rs`, the 10-condition gate); the
owner→agent control pattern (`active` switch / `desiredModels` — console writes the
record + `control_changed` nudge + agent re-reads + CAS-swap republish with
`preserve_console_fields`); APNs code-identity (0.9.18).

**Missing (the 0.9.19 work):**
1. `tier` (and the new `desiredTier`) are **not** in the Rust `ProviderRecord` struct
   (`pds.rs`) — the agent never publishes `tier` on the provider record.
2. `trustLevel` is **hardcoded `self-attested`** (`main.rs:669, 848`) — never flips to
   hardware-attested even with a bound MDA chain.
3. No owner-written opt-in field; the advisor's `enforceCodeAttestation` is a **global
   toggle** (reads as a fleet switch).
4. No optional in-app "upgrade" entry with honest tier labels.

## Workstreams

### WS1 — Lexicon (additive, lands first)
- `provider.json`: add optional **`desiredTier`** (owner-written intent; knownValues
  `best-effort` | `attested-confidential`), mirroring `desiredModels`. `tier` +
  `trustLevel` already exist. Regenerate types.

### WS2 — Agent (Rust): publish earned trust + reconcile toward `desiredTier`
- `pds.rs`: add `tier: Option<String>` + `desiredTier: Option<String>` to
  `ProviderRecord`.
- Derive + publish the **achieved** state from evidence (never self-declared):
  `trustLevel = hardware-attested` iff the attestation carries a bound MDA chain;
  `tier` = the attestation's computed tier. Replace the hardcoded `SelfAttested`.
- Read `desiredTier` (owner intent) on serve start + the existing `active` poll +
  `control_changed` nudge. If `attested-confidential` AND the build supports it
  (native feature + can load the native engine), enable the native engine and
  reconcile; if it can't (e.g. a non-native build), stay best-effort and surface a
  clear, content-safe reason (reuse `engineFault`-style messaging). Never break
  serving.
- Add `desiredTier` to `preserve_console_fields`.

### WS3 — Advisor: per-machine earned trust, not a fleet switch
- `registry.ts`: `confidentialEligible` **always** requires `codeAttested` (drop the
  `!enforceCodeAttestation ||` escape). Remove the global toggle.
- `main.ts`: construct the registry without the `apnsConfig !== null` flag. APNs being
  configured = the challenge *capability* is available; it is not an on/off for the
  fleet. A machine earns confidential only by passing the challenge — per machine.

### WS4 — UI: optional upgrade entry + honest labels
- Tray (`StatusView` / menu): an "Upgrade security…" entry. The MDA/hardware step
  reuses the existing `SecureModeWizard`; the tier opt-in writes `desiredTier`. Label
  the rungs honestly: hardware-attested = "genuine Mac, SIP verified — the operator
  can still read prompts"; confidential = "the operator cannot read your prompt".
- Console (machines dashboard): a per-machine optional "Upgrade security" action that
  writes `desiredTier` via the existing `setProviderRecordActive` PDS pattern +
  bridge mirror + advisor `control_changed` nudge. Honest tier labels on the badge.

### WS5 — Verify + bump
- Cross-stack typecheck/build/tests; bump to 0.9.19 (3 places + Cargo.lock); a
  non-native machine setting `desiredTier=attested-confidential` stays best-effort
  with a clear reason (no break); a hardware-attested machine publishes
  `trustLevel=hardware-attested`.

## Non-goals / invariants
- Never auto-upgrade; never break a self-attested machine.
- `trustLevel`/`tier` on the record are **evidence-derived claims** the requester
  re-verifies — the opt-in changes intent + triggers the work, it never fakes the
  achieved state.
- No global enforcement event; the per-machine record is the source of truth.
